# 加算マネージャー 設計哲学

> 最終更新: 2026-04-11
> 背景: デイほっと加算分析（2026-04-10）で得られた知見を製品設計に昇華

## なぜこの文書があるか

介護事業所の加算チェッカー／マネージャーは、単なる「要件チェックリスト」では価値を出せない。
現場では以下のような「思い込み」で加算を取り損ねている：

- 口腔機能向上加算には歯科衛生士が必要 → **実は看護職員でも可**
- 機能訓練指導員はPTでなければならない → **看護師・柔整師・あマ指師でも可**
- 管理栄養士は常勤配置必須 → **外部連携（栄養ケア・ステーション等）で可**
- 入浴介助加算Ⅱの居宅訪問はリハ職 → **介護福祉士でも可**
- サービス提供体制強化加算Ⅰは介護福祉士70%必須 → **勤続10年以上介護福祉士25%でも可**

これらは全て「OR条件」「外部連携ルート」「要件緩和パターン」で、法令通知には明記されているが、
現場では見落とされている。加算マネージャーの本質的な価値は、この**取りこぼしを機械的に発見する**ことにある。

## 5つの設計原則

### 原則1: 必須要件ではなく「満たせるルートを全探索」する

❌ **ダメな設計**
```
if 事業所.PT.count >= 1:
    機能訓練指導員要件 = True
else:
    機能訓練指導員要件 = False
```

✅ **正しい設計**
```
機能訓練指導員要件 = 
  事業所.PT.count >= 1 OR
  事業所.OT.count >= 1 OR
  事業所.ST.count >= 1 OR
  事業所.看護師.count >= 1 OR
  事業所.柔道整復師.count >= 1 OR
  事業所.あん摩マッサージ指圧師.count >= 1 OR
  事業所.鍼灸師.count >= 1
```

加算要件は法令上ほぼ全てOR条件・AND条件の論理式として表現できる。
チェッカーはこの論理式を**真偽値だけでなく、どのルートで満たされたか**まで返さなければならない。

### 原則2: 「取れない」で終わらせず、差分提案する

❌ **ダメな診断結果**
> サービス提供体制強化加算Ⅰ: 取得不可（介護福祉士比率 65% < 70%）

✅ **正しい診断結果**
> サービス提供体制強化加算Ⅰ: 現状未達
> ・条件A（介護福祉士70%）: 現在65%、あと2名の介護福祉士化で達成
> ・条件B（勤続10年以上介護福祉士25%）: 現在28% **→ 条件Bで達成済みの可能性あり**
> ・次アクション: 勤続年数を含む職員名簿を再確認してください
> ・達成時の増収: 月+約40,700円（年+約49万円）

チェッカーは「できない理由」ではなく「どうすればできるか」を返す。
複数ルートがある場合は**最短ルートを推奨**する。

### 原則3: 利用者ごとの加算ポートフォリオを最適化する

栄養アセスメント加算 ⇔ 栄養改善加算、口腔機能向上加算 ⇔ 口腔連携強化加算のように、
**同時算定不可だが利用者ごとに使い分け可能**な加算群が存在する。

これは単純なルールベースでは解けない、**最適化問題**である：

```
目的関数: 事業所の月間加算収入を最大化
制約:
  - 利用者 i について、加算 a と b は同時に算定できない
  - 加算 a は要件 R を満たす利用者のみ対象
  - 加算 b は要件 R' を満たす利用者のみ対象
決定変数: x[i][a] ∈ {0, 1}（利用者iに加算aを算定するか）
```

チェッカーは全利用者の状態を取り込み、**ポートフォリオ最適化**として解く必要がある。
実装は線形計画法（LP）でもよいし、制約が単純なら貪欲法でも良い。

### 原則4: 実データ（レセプト・職員マスタ）から自動検出する

加算マネージャーは、事業所が手入力したデータではなく、
**既に存在する業務データ**を入力として「取りこぼし」を自動検出する：

- **入力**: 介護給付費明細書PDF（レセプト）＋ 職員マスタ＋ 給与台帳
- **処理**: サービスコードから算定中加算を抽出 → 要件マッチング → 取りこぼし検出
- **出力**: 取得可能加算リスト＋ 準備アクション＋ 増収見込み

デイほっとの今回の分析フロー（PDF → サービスコード集計 → 要件マッチング → 差分提案）が
そのままチェッカーのコア処理になる。

### 原則5: 要件緩和の「抜け道」を知識として蓄積する

法令Q&Aや解釈通知には、文字通りには書かれていない「抜け道」がある：

- 中重度者ケア体制加算の「直近3か月」判定ルート（前年度実績を満たしていなくても算定開始可能）
- LIFE加算の段階的取得戦略
- 居宅訪問の共通化（個別機能訓練加算＋入浴介助加算Ⅱ＋生活機能向上連携加算Ⅱを1回の訪問で）

これらは**通知を読めば分かる**が、**現場では知られていない**。
チェッカーはこれらの知識をデータベース化し、該当するたびに提示する。

## 差別化戦略

競合の加算チェッカー（各介護ソフトベンダーの機能）との差別化ポイント：

| 項目 | 競合 | CareLinker加算マネージャー |
|------|------|---------------------------|
| 要件チェック | 真偽値のみ | ルート別・差分付き |
| 代替資格 | 主要資格のみ | 全OR条件を網羅 |
| 外部連携ルート | 未対応 | 栄養ケア・ステーション等を含む |
| 併算定最適化 | 単純警告のみ | 利用者ごとポートフォリオ最適化 |
| データ入力 | 手入力 | レセプトPDF自動取り込み |
| 要件緩和 | 通知文のリンクのみ | 抜け道を具体的アクションで提示 |
| 更新 | 改定時一括 | Q&A・事例を随時蓄積 |

## 知識グラフの構造（スキーマ要点）

加算マネージャーの心臓部は、介護保険全加算の要件を構造化データとして保持する知識グラフ。
最低限必要なノード型とフィールド：

### ノード: 加算（Kasan）
```yaml
kasan_id: kasan_tsusho_kouku_kinou_kojou_1
name: 口腔機能向上加算Ⅰ
service_type: 通所介護
unit: 150
unit_type: 単位/回
limit: 月2回まで
category: 口腔
life_required: false
new_in_2024: false
```

### ノード: 要件（Requirement）
```yaml
requirement_id: req_haichi_kouku
type: 配置要件 | 利用者割合 | 計画書 | LIFE提出 | 評価 | 体制
logic: OR  # AND / OR / NOT
children:
  - qualification: 言語聴覚士
  - qualification: 歯科衛生士
  - qualification: 看護職員  # ← 看護師 or 准看護師
```

### ノード: 資格（Qualification）
```yaml
qualification_id: qual_kangoshi
name: 看護職員
includes: [看護師, 准看護師]
substitutes_for:  # この資格で代替可能な他の資格要件
  - qual_kinou_kunren_shidouin
  - qual_kouku_kinou_kojou
external_ok: false  # 外部連携での配置が認められるか
```

### エッジ: 併算定不可（ExcludedWith）
```yaml
from: kasan_eiyou_assessment
to: kasan_eiyou_kaizen
scope: 同一月・同一利用者
usable_split: true  # 利用者ごとの使い分けは可能
```

### エッジ: 併算定減額（DiscountedWith）
```yaml
from: kasan_seikatsu_kinou_kojou_renkei_2
to: kasan_kobetsu_kinou_kunren
new_unit: 100  # 200→100に減額
```

### ノード: 要件緩和ルート（RelaxationRoute）
```yaml
route_id: route_chuujuudo_3months
applies_to: kasan_chuujuudo_care_taisei
condition: 前年度実績を満たしていない
alternative: 直近3か月間の利用者構成で判定可
action: ケアマネと連携し3か月集中で中重度者の受入を進める
```

### 事業所データ（Jigyosho）
```yaml
jigyosho_id: dayhot
name: デイサービスほっと
service_type: 通所介護
staff:
  - name: 稲垣真紀
    qualifications: [看護師]
    employment: 時給
  - name: 小菅美咲
    qualifications: [管理栄養士]
    employment: 時給
users: [ ... ]  # 要介護度・ADL・栄養リスク・口腔機能等
current_kasan: [kasan_kouku_assessment, kasan_saitei_kagaku, ...]
```

マッチング処理は「事業所.staff∪連携先」が「加算.要件.children」を満たすかを
**OR条件を展開して真偽値＋成立ルート**で返す。

## ユースケース #1: デイほっと（2026-04-10分析）

加算マネージャーの最初の実証例。レポート詳細は `operations/dayhot/dayhot_kasan_analysis.md`。

**入力**
- R8.2月分 介護給付費明細書PDF（97ページ）
- 職員マスタ（23名）、給与台帳、在籍資格情報

**処理結果（ダイジェスト）**
- 既取得加算8種を自動抽出（サ提体制Ⅱ、科学的介護、個別機能訓練Ⅰ(ロ)、処遇改善Ⅰ、栄養アセスメント、入浴Ⅰ 等）
- 取りこぼし加算: 7種
  - LIFE関連（個別機能訓練Ⅱ・ADL維持等Ⅰ）
  - 上位移行可能（入浴Ⅰ→Ⅱ、サ提体制Ⅱ→Ⅰ、栄養アセスメント→栄養改善への切替）
  - 新規取得可能（口腔機能向上Ⅰ、口腔連携強化）
- 検出した「思い込み」: 口腔機能向上加算の歯科衛生士必須、入浴Ⅱ居宅訪問のリハ職必須
- 最適化結果: 栄養・口腔は利用者ごとに使い分け→全員に何らかの加算を算定
- 増収見込: 月+約36万円（現体制のみ）、年+約432万円

**学び**
- レセプトPDFからサービスコードを集計するだけで既取得加算は正確に分かる
- 職員の資格情報を知識グラフと突き合わせると「看護師いるのに口腔未取得」のようなギャップが即検出できる
- 併算定不可ルールの使い分け最適化は手計算より機械的処理が圧倒的に早い
