# 使い方ガイド

加算マネージャー Web 版の使い方（事業所スタッフ・管理者向け）。

## 関連ドキュメント

- [CPOS PAT 接続ガイド](./CPOS_TOKEN.md) — CPOS と連携して分析する手順
- [データ取扱方針](./DATA_SAFETY.md) — PAT・個人情報・PDF の扱い
- [JSON 形式リファレンス](./json/) — アップロード可能な JSON ファイル仕様

---

## 1. ログイン / アクセス

カスタムドメイン（例: `https://kasan.example.jp`）にブラウザでアクセスします。

- 推奨ブラウザ: Google Chrome / Microsoft Edge / Firefox（いずれも最新版）
- スマホ・タブレットからでも閲覧できますが、PDF やデータをアップロードするためパソコンを推奨します
- 表示文字は 18px〜36px に拡大して設計されています。視認性が高い反面、横幅が狭い画面では一部レイアウトが折り返されます

ヘッダー右側の状態バッジで `Gemini 接続OK` が表示されていれば AI 補完分析が利用できます。`API キー未設定` 表示の場合は判定エンジンのみ利用可能です（管理者にお問い合わせください）。

## 2. 画面の流れ

### Step 1: 対象サービスを選ぶ

画面中央のセレクタから事業所のサービス種別を選択します。

| ドメイン | サービス | 状態 |
|---|---|---|
| 介護保険 | 通所介護 | ✅ 本番判定対応 |
| 介護保険 | 訪問介護 | ✅ 本番判定対応 |
| 介護保険 | 居宅介護支援 | ✅ 本番判定対応 |
| 介護保険 | 訪問看護（介護保険） | ✅ 本番判定対応 |
| 介護保険 | 小規模多機能型居宅介護 | β（AI 補完のみ） |
| 介護保険 | 特別養護老人ホーム | 準備中 |
| 医療保険 | 訪問看護（医療保険） | β（AI 補完のみ） |
| 障害福祉 | 居宅介護（障害福祉） | β（AI 補完のみ） |
| 障害福祉 | 就労継続支援 A 型 | β（AI 補完のみ） |
| 障害福祉 | 就労継続支援 B 型 | β（AI 補完のみ） |

### 状態の意味

| 表示 | 意味 |
|---|---|
| **（無印・本番判定対応）** | 厚労省告示・通知に基づき要件まで機械判定可。「判定エンジンのみ実行」「判定 + AI で分析する」の両方が高精度で動作 |
| **（β）** | 加算の名称・単位は登録済みだが、配置要件等の細部マスタ整備中。決定的判定は `情報不足` になりやすいが、Gemini 補完分析側で制度知識を使った提案は出る |
| **（準備中）** | マスタ未着手 |

`β` のサービスでも分析自体は実行できますが、結果は AI による参考提案が中心になります。最終確認は所管自治体・社労士にお願いします。

### Step 2: 事業所の情報を入力

| フィールド | 例 | 用途 |
|---|---|---|
| 事業所名 | デイサービスほっと | レポートの見出しに反映 |
| 事業所コード | DEMO-0004 | DEMO-XXXX を入れると公開デモ用テナントを参照 |
| 所在地・地域単価区分 | 東京都荒川区（6 級地） | 増収目安の補正・地域要件判定 |
| 職員構成 | 介護福祉士 4 名・看護師 1 名・機能訓練指導員は柔整師 | 配置要件 OR 条件の判定材料 |
| 利用者構成 | 利用者 40 名・要介護 3 以上 67%・認知症自立度 II 以上 35% | 利用者割合要件の判定材料 |
| 現在算定中の加算 | サービス提供体制強化加算 II・処遇改善加算 I | 上位区分への移行検討 |
| 気になっていること | 口腔機能向上加算が取れるか | AI への問い合わせ |

すべて任意入力です。空欄でも分析は実行できますが、入力するほど精度が上がります。

### Step 3: ファイルをアップロード（任意）

決定的判定エンジンに渡す 4 種類の専用枠があります。

| 枠 | 形式 | 役割 |
|---|---|---|
| 介護給付費明細書 PDF | `.pdf` | 算定中加算の自動抽出（要介護度分布・サービスコード・件数） |
| `tenant_status.json` | JSON | 事業所ステータス（要件確認の進捗） |
| `staff.json` | JSON | 職員集計（公開デモ用合成データ） |
| `user_summary.json` | JSON | 利用者集計（公開デモ用合成データ） |

その他の参考ファイル（追加資料）は最大 5 件・1 件 20 MB まで一括アップロードできます。

> ⚠️ **個人情報を含めないでください**: 被保険者番号・氏名・住所・電話番号は事前にマスキングしてください。本ツールはアップロードファイルを保存しません（メモリ上で展開・分析のみ実施）。

### Step 4: 分析を実行する

2 種類のボタンがあります。

| ボタン | 所要時間 | Gemini 利用 | 用途 |
|---|---|---|---|
| **判定 + AI で分析する** | 10〜30 秒 | あり | 取り方アクション・要件OR条件・代替ルートの提案 |
| **判定エンジンのみ実行** | 数秒 | なし | 加算別ステータス・不足証跡チェックリスト |

### Step 5（任意）: CPOS と連携して分析する

CPOS（介護記録・請求基盤）と連携できる環境では、最上部に **「🔐 CPOS と連携して分析する（任意）」** パネルが表示されます。

1. CPOS 管理者から個人 PAT（`cpos_pat_...`）を受け取る
2. パネルに CPOS URL と PAT を入力 → 「保存して接続確認」
3. 接続成功すると、自分の権限内の事業所が自動でセレクトに出る
4. 事業所と対象月を選んで「CPOS データで判定する」

PAT は加算マネージャの DB に保存されません。暗号化された HTTP-only Cookie だけがあなたのブラウザに残ります。

詳細手順は [CPOS_TOKEN.md](./CPOS_TOKEN.md)、データ扱いは [DATA_SAFETY.md](./DATA_SAFETY.md) を参照してください。

## 3. 結果の読み方

### 3-1. 決定的判定エンジン結果

#### サマリ集計

| アイコン | 意味 |
|---|---|
| ✅ 取得済/要件クリア | 全要件が確認済 |
| ⏸ 確認待ち | 一部の要件が `waiting`（証跡準備中） |
| ❌ 対象外/不可 | 要件未充足が確定 |
| ❔ 情報不足 | 判定材料が足りない（テナント情報や職員資料の登録待ち） |
| 💰 算定中 | レセプト PDF から算定中と推定 |

#### 加算別判定一覧

各加算ごとに以下の列が表示されます。

- **加算**: 名称と内部キー
- **判定**: 上記サマリのアイコン
- **単位**: `45 単位/日` / `100 単位/月` / `200 単位/回` / `所定単位×3%` のいずれか
- **DSL 評価**: 公式根拠確認済要件のみ機械評価した結果
  - `✅ clear`: 全要件達成
  - `❌ not_clear`: いずれかの要件が未達
  - `🟡 partially_clear`: 一部達成・一部証跡不足
  - `📭 不足証跡あり`: 評価可能だが証跡が足りない
  - `🔒 mapping 保留`: サービスコード照合未確認
  - `⏳ 根拠未確認 / ロジック未確認`: 評価対象外（マスタ整備中）
  - `🚫 対象外`: 公式根拠でこのサービス対象外と確定
- **達成ルート**: 満たされた条件のラベル（OR 条件のうちどれで達成したか）
- **不足証跡**: 判定で必要だが入力されていない fact path

#### 不足証跡チェックリスト

`📭 不足証跡あり` または `🔒 mapping 保留` の加算について、以下を整理して表示します。

- 優先度（高 / 中 / 低）
- 推奨確認資料
- 次のアクション（例: 「事業所内で職員の雇用契約書を確認」）

このチェックリストを埋めていくと、再度実行したときに判定が `clear` または `waiting` に進みます。

#### Markdown レポート

「Markdown レポートを表示」を開くと、9 セクション構成のテキスト版レポートが見られます。Notion / Confluence / メールにそのまま貼り付け可能です。

### 3-2. Gemini 補完分析（オプション）

判定エンジンが「unknown / waiting」と返した加算について、Gemini が以下を提案します。

- **サマリ**: 事業所の状況総評（300〜500 字）
- **増収見込み**: 月次・年次の概算（地域単価不明時は 10 円/単位として明示）
- **すぐ着手すべきアクション**: 優先順位付きで最大 5 件
  - タイトル / なぜ重要か / 具体的なステップ
- **取得候補の加算**: 各候補ごとに
  - ステータス（ready / waiting / blocked / unknown）
  - 単位・要件サマリ
  - 取り方のアクション
  - 追加で必要な情報
- **注意事項**: 法的根拠・地域差・改定タイミングの注意
- **前提条件**: AI が補った前提（地域単価未入力時は 10 円/単位など）

> Gemini の出力はあくまで提案です。算定可否の最終確認は所管自治体の指導課・社労士・行政書士にご相談ください。

## 4. データ出力

決定的判定の Markdown レポートはコピー & ペーストでドキュメント化できます。

API レスポンス（JSON）を直接欲しい場合は、ブラウザのデベロッパーツール `Network` タブで `/api/analyze` または `/api/judge` のレスポンスを確認するか、CLI（`npm run judge`）で同等の JSON を出力できます（[CLI マニュアル](./CLI.md) 参照）。

## 5. よくある質問

**Q. 「Gemini API キー未設定」と表示される**
A. Cloud Run 側に Gemini API キーが渡っていません。管理者に `.env` の更新と再デプロイ（`npm run deploy:cloudrun -- --skip-build`）を依頼してください（[デプロイマニュアル](./DEPLOYMENT.md) 参照）。

**Q. PDF をアップロードしても加算が検出されない**
A. レセプト帳票の形式（介護ソフトベンダー差異・自治体差異）により抽出精度が変動します。PDF 検出ゼロ ≠ 未算定 です。判定エンジンが `not_detected_in_pdf` を表示した場合は、要件マスタ側で取得候補があるかを確認してください。

**Q. 障害福祉（就労継続支援 B 型）の判定が「情報不足」ばかりになる**
A. 障害福祉のマスタは `draft` です。代表的な加算（処遇改善・福祉専門職員配置・目標工賃達成指導員・ピアサポート等）はマスタ化済みですが、要件論理式 DSL 未実装のため決定的判定では `unknown` になります。Gemini 補完分析側では制度知識を活用して提案を返します。

**Q. 同じ事業所コードで何度も実行できますか？**
A. はい。本ツールはサーバ側にデータを永続化していません。同じ入力で同じ結果が返り、毎回独立した分析として処理されます。

**Q. 古い API レスポンスを保存したい**
A. ブラウザの `右クリック → 名前を付けて保存` でページを保存するか、生レスポンス JSON をデベロッパーツールから保存してください。サーバ側にも履歴は残りません。

**Q. 個人情報を含めて分析させたい場合**
A. 本バージョンは公開 / 共有環境を想定しているため、個人情報を渡す設計にはしていません。事業所限定の閉じた環境で運用する場合は管理者に相談してください（VPC コネクタ・IAP 認証の追加が可能です。[技術マニュアル](./TECHNICAL.md) 参照）。

**Q. 「アクセス回数の上限に達しました」と表示された**
A. 短時間に大量のアクセスがあった場合に、IP 単位でレート制限が発動します。表示された秒数（通常 60 秒程度）をおいてから再度お試しください。継続的に発生する場合は管理者にお問い合わせください。

**Q. 「自動アクセスの可能性が高いと判定されました」と表示された**
A. Google reCAPTCHA v3 が bot 的なアクセス傾向を検知した場合に表示されます。通常はページを再読込して再度ボタンを押すと解消します。VPN や Tor 経由だとスコアが下がる傾向があります。

## 6. 用語集

| 用語 | 意味 |
|---|---|
| 加算 | 介護報酬・障害福祉サービス等報酬の上乗せ単位 |
| 単位 | 介護報酬の基本単位（1 単位 = 約 10 円、地域別単価あり） |
| 要件 | 加算を算定するために満たす必要のある条件（配置・体制・実績等） |
| 算定 | 加算を実際に請求すること |
| 取り漏れ | 要件を満たしているのに算定していない加算 |
| 解釈通知 | 厚生労働省が告示の解釈を補足する通知 |
| Q&A | 各都道府県・厚労省が公開する加算解釈の事例集 |
| 級地 | 地域単価の区分（1〜7 級地・その他、東京特別区は 1 級地） |
| evidence | レセプト PDF や届出書類などの根拠資料 |
| DSL | 要件論理式（OR / AND / ネスト条件）を JSON で表現する記述 |
| fact | DSL 評価で参照される事業所の事実情報（職員数・利用者比率など） |
